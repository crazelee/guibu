揭开binlog的神秘面纱

后端的同学肯定要接触MySQL，说到这个又得说主从，MHA架构等.这其中都有一个重要的东西，就是binlog，这个文件存储了MySQL的操作日志，主从也就是针对这个文件做的sql重放。之前一直知道这个，但没去细研究过，现如今遇到一个项目，都是基于binlog来做，提取binlog来做一些类似同步的功能，稍微研究了一下。

环境

为了方便测试，选择了mac 本地环境。
重启mysql：brew services restart mysql

查看binlog日志

如果没有开启binlog的，到网上找一些资料，还是比较容易设置的。

查看binlog都有哪些，这些是binlog的文件名

找到在磁盘的位置，我的在这个路径，brew安装的。
/usr/local/var/mysql/

简单做个测试

create database testbinlog;
use testbinlog;
create table test(`id` int(11) not null);
insert into test values(1);

可以看到这个binlog的文件已经变大了，这会来看看文件内容。

mysqlbinlog /usr/local/var/mysql/binlog.000036

你会发现一堆base64模样的东西

所以，为了能让人看懂

mysqlbinlog --base64-output=DECODE-ROWS -v /usr/local/var/mysql/binlog.000036



好，到这里你已经看到binlog里面的东西了。

【说明】binlog不记录查询的操作，只记录改删的操作。

恢复

现在我们来尝试做恢复，就是主从同步的事，咱进去把库删了。
先做一次binlog的备份。

cp /usr/local/var/mysql/binlog.000036 /usr/local/var/mysql/binlog.backup

删库

恢复
mysqlbinlog /usr/local/var/mysql/binlog.backup| mysql -uroot -p

验证
又回啦！！！

测试异常情况：做了第二次恢复，会提示失败，数据库已经存在。

指定时间范围
mysqlbinlog --base64-output=DECODE-ROWS -v --start-datetime='2019-01-15 00:00:00' --stop-datetime='2019-01-16 00:00:00' /usr/local/var/mysql/binlog.000029 > /usr/local/var/mysql/binlog.sql

package lib

import (
    "errors"

func (u Util)GenKey(day string) (string, error)  {

        return "", errors.New("key不能为空")
    }

编写单测 util_test.go，单测文件名 *_test.go

import (

复合数据类型

数组

数组长度是固定的。

var a [3]int  // array of 3 integers
var b [3]int = [3]int{1,2,3}
c := [...]int{1, 2, 3}

如果两个数组的所有元素都相等，数组相等

a := [2]int{1, 2}
b := [...]int{1,2}
fmt.Println(a==b)  // true

切片 Slice

变长的序列，每个元素有相同的类型。
一般写作[]T，Slice的语法和数组很像，只是没有固定长度而已。

一个slice由三部分构成：指针、长度、容量。内置的len和cap函数返回对应长度和容量。
len(s) == 0判断是否为空，不应该用 s == nil

哈希表 Map

args := make(map[string][int])  // 创建key为string，value为int型


# 初始化 1
ages := map[string][int]{
    "li" : 31,
    "bob" : 23,
}

相当于

# 初始化 2
ages := make(map[string][int])
ages["li"] = 31
ages["bob"] = 23

delete删除元素

delete(ages, "li") // remove element ages["li"]

把忽略value的map当做一个字符串集合，map[string]bool

程序结构

变量命名

名字的开头字母的大小写决定了名字在包外的可见性。
如果一个名字是大写字母开头的，那么它将是导出的，也就是说可以被外部的包访问。
例如fmt包的Printf函数就是导出的，可以在fmt包外部访问。包本身的名字一般总是用小写字母。

声明

有四种类型的声明语句：
变量var、常量const、类型type、函数实体对象func。

变量

1. var声明

var 变量名字 类型 = 表达式

一般 “类型”、或者“=”可以省略一个，例如：

var i, j, k int // int, int, int
var b, f, s = true, 2.3, "four" // bool, float64, string

用处：用于需要显式指定变量类型地方，或者因为变量稍后会被重新赋值而初
始值无关紧要的地方。

2. 简短声明

变量名字 := 表达式

anim := gif.GIF{LoopCount: nframes}
freq := rand.Float64() * 3.0
t := 0.0

用处：用于大部分的局部变量的声明和初始化。

注意，简短变量声明语句中必须至少要声明一个新的变量，下面的代码将不能编译通过：

f, err := os.Open(infile)
// ...
f, err := os.Create(outfile) // compile error: no new variables

指针

一个变量对应一个保存了变量对应类型值的内存空间。
一个指针的值是另一个变量的地址。

类型

类型声明语句一般出现在包一级，因此如果新创建的类型名字的首字符大写，则在外部包也可以使用。

type 类型名字 底层类型

Hello,GO

第一个go程序：hello.go

package main

import "fmt"

func main(){
    fmt.Println("Hello, Go")
}

go run 一次性调试

$ go run hello.go
Hello, Go

说明
1. Go是一门编译型语言，Go语言的工具链将源代码及其依赖转换成计算机的机器指令。
2. go run 编译一个或多个以.go结尾的源文件，链接库文件，并运行最终生成的可执行文件。

go build 直接生成可执行程序

如果不只是一次性实验，你肯定希望能够编译这个程序，保存编译结果以备将来之用。可以用build子命令，将直接生成可执行文件 hello，之后直接运行hello可执行文件，结果与go run时是一致的：

$ go build hello.go
$ ./hello
Hello, GO

基本认知

  1. Go语言的代码通过包（package）组织，包类似于其它语言里的库（libraries）或者模块（modules）。每个源文件都以一条 package 声明语句开始，这个例子里就是 package main , 表示该文件属于哪个包，紧跟着一系列导入（import）的包，之后是存储在这个文件里的程序语句。

  2. main 包比较特殊。它定义了一个独立可执行的程序，而不是一个库。在 main 里的 main 函数也很特殊，它是整个程序执行时的入口

  3. import 声明必须跟在文件的 package 声明之后。

  4. gofmt 工具把代码格式化为标准格式，并且 go 工具中的 fmt 子命令会对指定包, 否则默认为当前目录, 中所有.go源文件应用 gofmt 命令。

  5. 按照惯例，最好在每个包的包声明前添加注释；

  6. 空标识符（_下划线）类似垃圾箱，将不需要的变量丢弃，如下，只需要range的value，而不需要索引，则可以赋值给他丢弃

for _, arg := range os.Args[1:] {
s += sep + arg
sep = " "
}

RESTful API 介绍

Go 语言中常用的 API 风格是 RPC 和 REST，常用的媒体类型是 JSON、XML 和 Protobuf。在 Go API 开发中常用的组合是 gRPC + Protobuf 和 REST + JSON。

REST 代表表现层状态转移（REpresentational State Transfer）

RPC （Remote Procedure Call，RPC）该协议允许运行于一台计算机的程序调用另一台计算机的子程序，而程序员无须额外地为这个交互作用编程。

RPC 相比 REST 的优点主要有 3 点：

RPC+Protobuf 采用的是 TCP 做传输协议，REST 直接使用 HTTP 做应用层协议，这种区别导致 REST 在调用性能上会比 RPC+Protobuf 低
RPC 不像 REST 那样，每一个操作都要抽象成对资源的增删改查，在实际开发中，有很多操作很难抽象成资源，比如登录操作。所以在实际开发中并不能严格按照 REST 规范来写 API，RPC 就不存在这个问题
RPC 屏蔽网络细节、易用，和本地调用类似...

REST 相较 RPC 也有很多优势：

轻量级，简单易用，维护性和扩展性都比较好
REST 相对更规范，更标准，更通用，无论哪种语言都支持 HTTP 协议，可以对接外部很多系统，只要满足 HTTP 调用即可，更适合对外，RPC 会有语言限制，不同语言的 RPC 调用起来很麻烦
JSON 格式可读性更强，开发调试都很方便
在开发过程中，如果严格按照 REST 规范来写 API，API 看起来更清晰，更容易被大家理解...

内部系统之间调用用 RPC，对外用 REST

一、分布式微服务设计原理

  • 每个职责单一的功能放在一个独立的服务中
  • 每个服务运行在单独的进程中
  • 每个服务有多个实例在运行，容器化，达到平滑伸缩
  • 每个独享数据存储，数据库、缓存、队列等
  • 每个服务高度自治
  • 每个服务可以根据性能需求进行水平伸缩
